# Slack 約會議自動排程

## 概述

此 skill 為**手動觸發**，當使用者說「排程」、「幫我跑一次」或類似指令時執行。每次執行會搜尋所有尚未建立過 Calendar 的約會議訊息，自動同步到 Google Calendar，並視情況在 Notion 建立會議記錄。

**完整流程：**
1. 讀取上次執行時間（state 追蹤）
2. 搜尋 Slack 中你最近發送、包含各會議類型關鍵字的訊息（含改約訊息）
3. 判斷每則訊息是「新增約會」還是「改約/改時間」
4. 解析每則訊息的專案名稱、日期、時間、會議類型
5. **新增約會**：在 Google Calendar 建立活動（無時間時活動名稱加 `tbc`）
   **改約**：在 Google Calendar 找到原本的活動並更新時間
6. 若類型為「提案」或「工作會議」（且不含 A/B copy / final），在 Notion 建立會議記錄頁面
7. 更新 state，輸出執行摘要

---

## 所需 Connectors

執行前請確認以下 connectors 已連接：
- **Slack** — 搜尋你發送的訊息
- **Google Calendar** — 建立活動（gcal_* 工具已內建）
- **Notion** — 建立會議記錄（僅部分類型需要）

## Step 1：搜尋 Slack 訊息（多關鍵字搜尋）

使用者的訊息用詞多樣，不會只寫「約會議」。因此使用 **多次搜尋** 涵蓋所有可能的會議關鍵字，再合併去重。

對以下每個 query，各執行一次 `slack_search_public_and_private`：

| 搜尋目標 | Query |
|---------|-------|
| 提案相關 | `提案 from:<@U095B7PCDTL>` |
| A copy 相關 | `A copy from:<@U095B7PCDTL>` |
| B copy 相關 | `B copy from:<@U095B7PCDTL>` |
| final 相關 | `final from:<@U095B7PCDTL>` |
| 工作會議相關 | `工作會議 from:<@U095B7PCDTL>` |
| 進場相關 | `進場 from:<@U095B7PCDTL>` |
| 一般約會議 | `約會議 from:<@U095B7PCDTL>` |
| 改約相關 | `改約 from:<@U095B7PCDTL>` |
| 改時間相關 | `改時間 from:<@U095B7PCDTL>` |

每次搜尋參數：`sort: "timestamp"`, `sort_dir: "asc"`

合併所有搜尋結果，**以 message ts 去重**，過濾出 ts > `last_processed_ts` 的訊息。

### 過濾與分類：區分「新約」和「改約」

合併後，先判斷每則訊息屬於哪一種，再分流處理：

**🔁 改約訊息（優先判斷）**：訊息中包含以下任一詞 — `改約`、`改時間`、`會議改`、`改到`、`改成`、`移到`、`延到`、`取消改`，**且**有可解析的新日期/時間 → 進入改約流程（見 Step 3b）

**✅ 新約訊息**：同時符合以下兩條件 →
1. **有約會意圖**：包含 `約`、`安排`、`排一下`、`排個`、`討論`、`開`（搭配會議類型使用）
2. **有可解析的日期或時間**：包含日期（數字日期、明天/後天/下週等）或時間（幾點、AM/PM）

若以上兩類都不符合，跳過該訊息。若無任何有效訊息，輸出「✅ 無新的約會議訊息，結束本次執行。」並跳至 Step 5 更新 state。

## Step 2：解析每則訊息

對每則通過過濾的訊息，執行以下解析。

### 2a. 專案名稱（從頻道名稱取得）

1. 取得訊息所在的頻道名稱
2. 去除開頭的年份（格式 `YYYY` 或 `YYYY ` 或 `YYYY-`），取剩餘部分為專案名稱
3. 去除多餘空白

| 頻道名稱 | 專案名稱 |
|---------|---------|
| `2026 台灣設計展` | `台灣設計展` |
| `2025金融廣告` | `金融廣告` |
| `2026-環境部形象片` | `環境部形象片` |

### 2b. 日期與時間

從訊息內容中識別日期與時間表達式：

**日期格式（常見）：**
- 絕對日期：`3/20`、`3月20日`、`03/20`
- 相對日期：`今天`、`明天`、`後天`

**相對週次日期計算規則（重要）：**
計算「下週X」或「這週X」時，請嚴格按照以下邏輯：
- **今天是星期幾** → 找到當週的**週一**
- **這週X** = 本週週一 + (X - 1) 天
- **下週X** = 本週週一 + 7 + (X - 1) 天

星期對應天數：一=1, 二=2, 三=3, 四=4, 五=5, 六=6, 日=7

範例（今天 = 週五 2026-03-13）：
- 今天 = 週五 → 本週週一 = 2026-03-09
- `下週三` = 2026-03-09 + 7 + 2 = **2026-03-18**（週三）✓
- `下週一` = 2026-03-09 + 7 + 0 = **2026-03-16**（週一）✓

**時間格式（常見）：**
- `下午2點`、`早上10點`、`上午9點半`
- `14:00`、`10:30`、`9:00`
- `下午2點半`（即 14:30）

**預設值：**
- 沒有時間：預設 `09:00`，並標記 **`has_time = false`**（活動名稱將加上 `tbc`）
- 有明確時間（幾點、AM/PM、數字時間）：標記 **`has_time = true`**
- 沒有日期：預設今天
- 活動長度：預設 **1 小時**
- 時區：**Asia/Taipei（UTC+8）**

### 2c. 會議類型（動詞＋名詞組合優先）

**偵測邏輯：先找「動作動詞 + 類型」的組合，再找單獨的類型關鍵字。**

這樣能區分「*開* 工作會議」（類型）和「討論*提案*方向」（話題）。

**Step 1 — 在訊息中尋找以下完整動作組合（任一符合即確定類型）：**

| 動作組合 | 偵測到的類型 |
|---------|------------|
| `約開工作會議`、`開工作會議`、`約工作會議` | 工作會議 |
| `約看 A copy`、`看 A copy`、`約 A copy` | A copy |
| `約看 B copy`、`看 B copy`、`約 B copy` | B copy |
| `約看 final`、`看 final`、`約 final` | final |
| `約提案`、`約做提案`、`提案會議` | 提案 |
| `約進場`、`進場會議` | 進場會議 |
| `約開會`、`約會議` | 一般會議 |

**Step 2 — 若 Step 1 無結果，再用單關鍵字掃描（優先順序）：**

| 優先順序 | 類型 | 關鍵字 |
|---------|------|--------|
| 1 | A copy | `A copy`、`a copy`、`Acopy` |
| 2 | B copy | `B copy`、`b copy`、`Bcopy` |
| 3 | final | `final`、`Final`、`FINAL` |
| 4 | 提案 | `提案` |
| 5 | 工作會議 | `工作會議` |
| 6 | 進場會議 | `進場會議`、`進場` |
| 7 | 一般會議 | 以上皆無 |

> 注意：A/B copy 和 final 在 Step 2 優先順序調高，因為單獨出現時歧義較少；「提案」則因為常作為話題詞出現（如「討論提案方向」），所以在 Step 2 降低優先順序，避免誤判。

## Step 2.5：出勤衝突檢查

在建立 Calendar 活動前，先確認這個日期沒有人請假而導致會議無法成立。

### 2.5a. 讀取「出勤近況報告」頻道

使用 `slack_search_public_and_private`，搜尋 `請假 in:出勤近況報告`，取得近期請假訊息。

對每則訊息，記錄：
- **請假人**：訊息的發送者（sender 的 user_id / 顯示名稱）
- **請假日期**：從訊息內文中解析，支援以下格式：
  - `4/15 請假` → 單日 4/15
  - `4/15-4/17 請假` → 連續區間 4/15 到 4/17（含頭尾）
  - `4/15、4/16 請假` → 多個單日
  - 相對日期（`明天`、`下週一` 等）→ 同 Step 2b 的計算方式

### 2.5b. 比對會議日期

將本次解析到的**會議日期**，與所有請假記錄逐一比對：

- **無衝突** → 直接進入 Step 3a，正常建立 ✅
- **有人在該日請假** → 進入 2.5c

### 2.5c. 確認出席人員

從**原始約會議訊息**中判斷誰要出席：

| 訊息中的標記方式 | 出席人員 |
|----------------|---------|
| 有 `@具體成員`（如 `@小明 @Annie`） | 僅被標記的人 |
| 有 `@channel` 或 `@here` | 頻道所有成員 |
| 沒有任何 @標記 | 預設為頻道所有成員 |

若需取得頻道全部成員，使用 `slack_read_channel` 取得成員列表。

### 2.5d. 衝突判斷與行動

**狀況 A：請假者不在出席名單 → 會議可成立**
- 繼續 Step 3a 正常建立 Calendar ✅
- 在摘要中備註：`⚠️ [姓名] 當天請假，但不在出席名單中，不影響此會議`

**狀況 B：請假者在出席名單中 → 會議不可成立**
1. **不建立** Google Calendar 活動，也不建立 Notion 頁面
2. 使用 `slack_send_message` 發私訊給 Annie（`U095B7PCDTL`）：

```
⚠️ 會議衝突通知

「[專案名稱]_[類型]」無法在 [日期] 建立，因為：
・[請假人姓名] 在 [請假日期] 請假

請重新安排時間，我會在你約好後自動建立到 Calendar 😊
```

3. 在摘要中標記：`❌ 無法建立（[姓名] 當天請假，已通知 Annie）`

> **若搜尋「出勤近況報告」失敗或找不到任何訊息**：視為無衝突，繼續正常流程，並在摘要備註：`⚠️ 無法讀取出勤報告，已跳過衝突檢查`

## Step 3a：建立 Google Calendar 活動（新約）

使用 `gcal_create_event`。活動名稱規則：
- **有明確時間（`has_time = true`）**：`[專案名稱]_[類型]`
- **無時間（`has_time = false`）**：`[專案名稱]_[類型] tbc`（代表時間待確認）

```
summary 範例（有時間）：
  台灣設計展_工作會議
  金融廣告_A copy
  環境部形象片_提案

summary 範例（無時間，加 tbc）：
  台灣設計展_工作會議 tbc
  金融廣告_A copy tbc

start: { dateTime: <解析日期時間>, timeZone: "Asia/Taipei" }
end:   { dateTime: <開始 + 1 小時>, timeZone: "Asia/Taipei" }
calendarId: "peppercorns.media.art@gmail.com"
```

活動建立成功後，記錄 `eventId` 及 `htmlLink`。

## Step 3b：更新 Google Calendar 活動（改約）

當訊息為「改約/改時間」類型時，執行以下流程：

### 找到原本的活動

1. 從訊息中解析出**專案名稱**和**會議類型**（方法同 Step 2a / 2c）
2. 使用 `gcal_list_events` 搜尋 `peppercorns.media.art@gmail.com`，時間範圍設寬（例如前後 30 天）
3. 找出 summary 符合 `[專案名稱]_[類型]` 或 `[專案名稱]_[類型] tbc` 的活動
4. 若訊息中有提到「原本的日期」（例如「把 3/24 的提案改到 3/26」），優先找那個日期的活動

若找不到對應活動：輸出警告「⚠️ 找不到原本的 Calendar 活動，請手動調整。」，跳過此則訊息。

### 更新活動時間

1. 從訊息中解析出**新的日期/時間**
2. 使用 `gcal_update_event` 更新：
   - `start.dateTime`：新時間（Asia/Taipei）
   - `end.dateTime`：新時間 + 1 小時
   - `summary`：若新時間有明確時間（`has_time = true`），移除舊名稱結尾的 ` tbc`；若無時間，保留或加上 ` tbc`

```
calendarId: "peppercorns.media.art@gmail.com"
eventId: <找到的原活動 ID>
```

更新成功後記錄 `eventId` 及新的 `htmlLink`。

> 改約流程**不需要**建立新的 Notion 頁面（Step 4 跳過）。

## Step 4：建立 Notion 會議記錄（條件式）

**只在同時符合以下兩條件時執行：**
1. 會議類型是 `提案` 或 `工作會議`
2. 訊息中**沒有**出現 `A copy`、`B copy`、`final`（任何大小寫）

若條件不符，跳過本 Step，記錄「不需建立 Notion 頁面」。

### 4a. 在 Notion 搜尋專案

使用 `search` 搜尋專案名稱（例如「台灣設計展」），找到 Notion 中名為 **"Project"** 的 database 中對應的 page。

若找不到：輸出警告「⚠️ Notion 找不到專案：[專案名稱]，跳過 Notion 建立」，繼續執行。

### 4b. 找到「專案管理」下的 database

使用 `fetch` 取得該專案 page 的內容，找到名為「專案管理」的 block，取得其底下的 database ID。

### 4c. 建立會議記錄頁面

使用 `create-pages`，在該 database 中建立新頁面：

| Property | 設定值 |
|---------|--------|
| 工作細項（title）| 會議類型（例：工作會議、提案）— **不**含專案名稱 |
| 工作項目 | 會議紀錄 |
| 時程 | 解析到的日期與時間 |

> **關於 Template**：Notion API 目前無法直接透過 API 套用 template（「會議記錄樣板」）。建立頁面後，請在摘要中提醒使用者手動套用 template。

2. 輸出本次執行摘要，格式如下：

```

   📆 Google Calendar：已更新 ✅（tbc 已移除）
   📝 Notion：跳過（改約不建立新頁面）
```

## 錯誤處理

| 情況 | 處理方式 |
|-----|---------|
| 無法解析日期/時間 | 記錄警告，跳過該則訊息，繼續下一則 |
| Notion 找不到對應專案 | 記錄警告，Calendar 仍正常建立 |
| Slack connector 未連接 | 輸出錯誤「Slack 未連接，請先連接 Slack connector」後停止 |
| Calendar 建立失敗 | 記錄錯誤，繼續處理下一則訊息 |
| 改約但找不到原活動 | 輸出警告「⚠️ 找不到原本的 Calendar 活動，請手動調整」，跳過該則 |
| 出勤近況報告搜尋失敗 | 視為無衝突，繼續正常流程，在摘要備註「⚠️ 無法讀取出勤報告，已跳過衝突檢查」 |
